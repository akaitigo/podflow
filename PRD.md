# PRD: podflow

**作成者**: Ryusei | **日付**: 2026-03-27 | **ステータス**: ドラフト
**アイデアID**: #576 | **ドメイン**: Podcast制作 | **難易度**: Low

## 問題定義

Podcast制作は企画・ゲスト調整・収録・編集・公開と多くのステージを経るが、各ステージが異なるツール（Google Docs、メール、DAW、各配信プラットフォーム）に散在しており、進捗管理が属人化しやすい。特に複数エピソードを並行して制作する場合、どのエピソードがどのステージにあるかの把握が困難になる。

podflowは制作ワークフローをカンバンで一元管理し、音声ファイルからショーノート・SNS告知文の自動生成、複数配信プラットフォームへの一括公開までを統合する。

## 目標と成功指標

| 目標 | 指標 | ターゲット |
|------|------|----------|
| エピソード管理の一元化 | カンバンで全エピソードのステータスが可視化されている | 5エピソード以上を同時管理可能 |
| 制作時間の短縮 | ショーノート作成時間 | 手動比50%削減 |
| 配信の効率化 | 複数プラットフォームへの公開操作 | 1クリックで一括配信 |
| ゲスト調整の省力化 | ゲストとの日程調整にかかるやりとり回数 | 3回以内で確定 |

## スコープ外（MVP）

- リアルタイム音声編集（DAW機能）
- 収益管理・アナリティクスダッシュボード
- チーム間のリアルタイムコラボレーション
- 動画Podcast対応
- モバイルアプリ

## ユーザーストーリー

1. **Podcastホスト**として、エピソードの制作進捗をカンバンボードで確認したい。なぜなら複数エピソードの並行制作で抜け漏れを防ぎたいから。
2. **Podcastホスト**として、収録音声をアップロードしてショーノートを自動生成したい。なぜなら毎回1-2時間かかる手作業を省きたいから。
3. **Podcastホスト**として、完成エピソードをSpotify・Apple Podcasts・YouTubeに一括公開したい。なぜなら各プラットフォームに個別ログインして投稿する手間を省きたいから。
4. **Podcastプロデューサー**として、ゲストの出演スケジュールと連絡先を管理したい。なぜなら調整メールの見落としを防ぎたいから。

## 技術要件

### 技術スタック

- **フロントエンド**: TypeScript / React
- **バックエンド**: Kotlin / Quarkus
- **通信**: gRPC (Protocol Buffers)
- **データベース**: PostgreSQL
- **インフラ**: GCP Cloud Run, Cloud Storage

### アーキテクチャ

```
[React SPA] --gRPC-Web--> [Quarkus API] ---> [PostgreSQL]
                                |
                          [Cloud Storage]
                          (音声ファイル)
```

モノレポ構成:
- `frontend/` — React SPA (Vite + TypeScript)
- `backend/` — Quarkus API (Kotlin + gRPC)
- `proto/` — Protocol Buffers 定義（共有）

### 非機能要件

- パフォーマンス: API応答 < 200ms (p95)
- セキュリティ: 認証はJWT、音声ファイルは署名付きURL
- スケーラビリティ: 初期100ユーザー対応

## 競合分析

| 競合 | 特徴 | ギャップ |
|------|------|---------|
| Riverside.fm | 高品質リモート収録 + 編集 | ワークフロー管理なし、配信は手動 |
| Descript | 文字起こしベース編集 | 制作管理ではなく編集特化 |
| Podbean | ホスティング + 基本アナリティクス | カンバン管理なし、ゲスト調整なし |
| Notion/Trello | 汎用カンバン | Podcast特化機能なし（配信、ショーノート） |

**差別化ポイント**: Podcast制作に特化したカンバン + 音声→ショーノート→SNS告知の自動生成パイプライン + 複数プラットフォーム一括配信の統合。汎用ツールの組み合わせでは実現できない一気通貫のワークフロー。

## 技術リスク

| リスク | 発生可能性 | 影響 | 軽減策 |
|--------|----------|------|--------|
| 文字起こしAPI精度が日本語で不十分 | 中 | 高 | Whisper API + 後処理パイプラインで補正。MVPでは外部API依存 |
| gRPC-Web のブラウザ互換性 | 低 | 中 | Envoy proxy または Connect protocol で対応 |
| 各配信プラットフォームAPIの制約 | 中 | 中 | MVPではRSSフィード生成のみ。直接API連携はPhase 2 |

## マイルストーン

### Phase 0: MVP（2週間）

1. プロジェクト基盤（CI/CD、lint、テスト）
2. エピソード CRUD + カンバンUI
3. gRPC API + PostgreSQL スキーマ
4. 基本的なフロントエンド（カンバンボード）
5. E2Eテスト + README

### Phase 1: コア機能（Phase 0完了後）

- 音声アップロード + Cloud Storage 連携
- 文字起こし→ショーノート自動生成
- RSSフィード生成

### Phase 2: 配信統合

- Spotify/Apple Podcasts/YouTube 連携
- ゲスト管理・日程調整
