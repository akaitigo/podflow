# ADR-001: 認証・認可を MVP スコープ外とする

## ステータス

Superseded by [ADR-005](005-jwt-authentication.md)

## コンテキスト

podflow はポッドキャスト制作ワークフローを管理するツールである。本番運用にあたり認証・認可は必須だが、MVP フェーズでは以下の理由からスコープ外とする判断が必要になった。

### 検討した選択肢

1. **Firebase Authentication**: Google エコシステムとの親和性が高い。フロントエンド SDK が充実。
2. **Keycloak**: 自前ホスティングが必要だがカスタマイズ性が高い。
3. **Auth0 / Clerk**: マネージドサービスで導入が容易だが、コスト発生。
4. **MVP フェーズでは実装しない**: 開発速度を優先し、認証は Phase 1 で対応。

## 決定

**認証・認可は MVP スコープ外とし、Phase 1 で対応する。**

理由:

1. **開発速度の優先**: MVP はワークフロー管理の UX 検証が目的であり、認証の実装に時間を割くことは目的に反する。
2. **シングルユーザー前提**: MVP は開発者本人のみが使用する想定であり、マルチテナントの認証は不要。
3. **後付けが可能**: gRPC インターセプターで認証を後から追加できるアーキテクチャになっている。Quarkus の `quarkus-oidc` 拡張で OIDC/JWT 認証を導入可能。
4. **セキュリティリスクの限定**: MVP はローカル/プライベート環境での使用を想定。パブリック公開時には必ず認証を実装する。

## リスク

- MVP をそのまま公開環境にデプロイすると、認証なしで全操作が可能になる。
  - **軽減策**: README およびデプロイドキュメントに「本番公開前に認証実装が必須」と明記する。
  - **軽減策**: Phase 1 の最優先タスクとして認証を位置づける。

## 結果

- MVP では認証・認可を実装しない
- Phase 1 で Firebase Authentication または OIDC ベースの認証を導入予定
- gRPC インターセプターで認証チェックを挟む設計を採用予定
- デプロイ前チェックリストに「認証実装済み」を追加
